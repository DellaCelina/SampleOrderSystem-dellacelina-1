# REQUIREMENT.md

## 배경 (Background)

현재 저장소에는 비어 있는 Visual Studio C++ 콘솔 프로젝트 뼈대만 존재한다
(`SampleOrderSystem/SampleOrderSystem.vcxproj`, `SampleOrderSystem.slnx`) — 아직 애플리케이션
소스 코드는 없다. `docs/PREREQUIREMENT.md.txt`와 `[CRA_AI] Day3_개인과제_반도체시료관리.pdf`가
"반도체 시료 생산주문관리 시스템"("S-Semi")이라는 콘솔 애플리케이션의 기능 명세이며, 기존에
스프레드시트/메모장 등으로 수작업 처리하던 반도체 시료 주문 관리 프로세스를 대체하는 것이 목적이다.

이 시스템에는 두 가지 역할이 있다: 고객의 시료 요청을 등록하고 확정된 주문을 출고하는
**주문 담당자**, 그리고 주문을 승인/거절하고 재고 부족분에 대한 FIFO 생산 라인을 관리하는
**생산 담당자**. 두 역할 모두 콘솔 앱을 실행하는 동일한 사용자가 수행하며, 별도의 로그인/권한
분리는 없다 — 콘솔은 두 기능에 대한 메뉴를 모두 노출할 뿐이다. `CLAUDE.md`에 따라, 아키텍처는
세 개의 자매 PoC 저장소의 패턴을 따른다: `ConsoleMVC-dellacelina-1`(MVC 콘솔 구조),
`DataPersistence-dellacelina-1`(JSON 파싱/파일 영속화), `DataMonitor-dellacelina-1`(JSON을
DB처럼 다루는 데이터 관리), `DummyDataGenerator-dellacelina-1`(스키마 기반 더미 데이터 생성).
이 요구사항 문서는 이 시스템의 최초 엔드투엔드 구현을 다룬다(확장할 이전 버전은 없음).

## 범위 (Scope)

- **시료 관리**: 시료를 등록한다(시료 ID, 이름, 평균 생산시간(정수 분 단위), 수율 0~1); 등록된
  모든 시료를 현재 재고와 함께 목록으로 조회한다; 시료 ID(정확히 일치) 또는 이름(대소문자
  구분 없는 부분 문자열 일치) 중 하나로 검색한다 — 그 외 검색 필드는 범위 밖.
- **주문 생애주기**: 시료 ID, 고객명, 수량으로 주문을 접수한다(`RESERVED`), 주문번호는
  `ORD-####` 형식(4자리 순번, 아래 롤오버 제약 참고)으로 발급된다; 승인 대기 중인
  `RESERVED` 주문 목록을 조회한다; 주문을 승인한다(미선점 재고가 전체 수량을 충당하면
  `CONFIRMED`로 자동 전환, 부족하면 부족분을 계산하여 생산 큐에 등록하고 `PRODUCING`으로
  전환); 주문을 거절한다(`RESERVED` → `REJECTED`, 종결 상태); `CONFIRMED` 주문을 출고한다
  (`CONFIRMED` → `RELEASED`, 출고 수량만큼 재고 차감). 이번 이터레이션에서 주문 담당자는
  `RESERVED` 상태의 주문을 수정하거나 취소할 수 없다 — 비목표(Non-goals) 참고.
- **재고와 여유분의 구분**: 시료별 현재 재고를 추적한다; "미선점" 여유분은 재고에서 해당
  시료의 다른 `PRODUCING`/`CONFIRMED` 주문이 이미 선점한 수량을 뺀 값으로 계산한다; 승인
  판단과 모니터링 화면 모두 두 수치를 구분하여 표시한다.
- **생산 큐**: 단일 FIFO 큐(병렬 생산 라인이 아닌 하나의 생산 라인)로, JSON 파일에 영속화된다.
  각 큐 엔트리는 최소한 주문번호, 시료 ID, 부족 수량, 실제 생산 수량
  (`ceil(부족분 / 수율)`), 예상 완료 시각을 저장한다. 생산 라인이 하나뿐이므로 각 엔트리는
  반드시 순차적으로 처리된다: 앞선 엔트리가 완료되기 전에는 다음 엔트리의 생산이 시작될 수
  없다. 따라서 새로 등록되는 엔트리의 예상 완료 시각은 다음과 같다:
  `max(등록 시각, 직전 큐 엔트리의 예상 완료 시각(있는 경우)) + 평균 생산시간(분) * 실제 생산
  수량`. 큐가 비어 있던 상태에서의 첫 엔트리는 `등록 시각 + 평균 생산시간 * 실제 생산 수량`로
  단순화된다.
- **지연(lazy) 정산**: 주문 상태, 생산 라인 상태, 재고 중 어느 것을 조회하든 먼저 생산 큐
  파일을 훑어 완료 시각이 지난 엔트리를 찾아 제거하고, 그에 대응하는 주문 상태
  (`PRODUCING` → `CONFIRMED`)와 재고 수준을 갱신한 뒤 조회 결과를 반환한다. 백그라운드
  스레드/타이머는 없다. "현재 시각"과의 모든 비교는 실제 벽시계 API를 직접 호출하는 대신
  단일한 주입 가능 시계(clock) 추상화(제약 사항 참고)를 거치므로, 테스트에서 시간을
  결정적으로 앞당기거나 고정할 수 있다.
- **모니터링**: 상태별 주문 수(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASED`, `REJECTED`는
  제외); 시료별 재고 수준과 고갈 여부 상태 라벨(수락 기준 참고 — 더 세분화된 여유/부족 임계값은
  이 요구사항에서 명시적으로 보류됨).
- **생산 라인 조회**: 현재 생산 중인 하나의 엔트리(큐가 비어있지 않은 경우)와 그 뒤에 대기
  중인 나머지 FIFO 큐를, 각각 시료·부족분·실제 생산 수량·예상 완료 시각과 함께 보여준다.
  아래 비목표와 일관되게, 어느 순간이든 "생산 중"인 엔트리는 최대 하나뿐이다.
- **콘솔 MVC UI**: 시료 관리, 주문 접수, 주문 승인/거절, 모니터링, 생산 라인 조회, 출고
  처리를 위한 메인 메뉴와 하위 메뉴들로 구성하며, Model/View/Controller를 분리한다.
- **JSON 영속화 계층**: 시료, 주문, 생산 큐를 각 "테이블"별로 명시적 스키마(필드명, 타입,
  제약)를 가진 JSON 파일로 저장하고 직접 읽고 쓴다(외부 DB 없음), 애플리케이션 재시작 후에도
  유지된다.
- **데이터 모니터링 도구**: `DataMonitor-dellacelina-1`의 방식을 따라, 현재 JSON 기반 데이터
  (시료/재고, 주문, 생산 큐)를 언제든 콘솔에서 확인할 수 있는 기능. 별도 빌드 타겟이 아니라
  동일한 `SampleOrderSystem` 실행 파일 안의 메뉴/모드다 — 제약 사항 참고.
- **더미 데이터 생성기**: `DummyDataGenerator-dellacelina-1`의 방식을 따라, 스키마에 부합하는
  랜덤 시료/주문 테스트 데이터를 생성해 JSON 저장소에 추가하는 도구. 이 역시 동일한
  `SampleOrderSystem` 실행 파일 안의 메뉴/모드(또는 커맨드라인 플래그)다 — 제약 사항 참고.

## 비목표 (Non-goals)

- 인증, 사용자 계정, 역할 기반 접근 제어 없음 — 주문 담당자와 생산 담당자는 별도의 로그인으로
  구분되지 않고, 동일한 콘솔 세션에서 동작한다.
- 네트워킹, 웹 UI, 다중 사용자/동시 프로세스 지원 없음 — 단일 콘솔 프로세스, 한 번에 한 명의
  사용자만 사용.
- 외부 데이터베이스(SQL/NoSQL 서버) 없음 — JSON 파일이 유일한 영속화 수단이다.
- 생산 완료를 위한 실제 백그라운드 스케줄러/타이머 스레드 없음 — 완료 처리는 명세된 대로
  조회 시점에만 지연 방식으로 해소된다.
- 생산 수율 손실의 시뮬레이션 없음 — 계산된 `ceil(부족분 / 수율)` 수량은 항상 100% 재고에
  더해진다(수율은 *생산 수량*에만 영향을 주고 생존율에는 영향을 주지 않음), 따라서 재고가
  필요량 이상으로 남을 수 있다; "실제" 수율 기반 손실 구현은 명시적으로 범위 밖이다.
- 새 주문이 같은 시료에 대해 들어와도 이미 큐에 들어간 생산 엔트리를 재계산/재조정하지
  않는다 — 각 큐 엔트리는 자신의 승인 시점에 계산된 부족분으로 고정된다.
- `RESERVED`를 벗어난 이후의 주문 취소/수정 없음(`CONFIRMED`/`PRODUCING` → `RESERVED`로의
  역전 없음, 부분 출고/부분 릴리스 없음).
- **`RESERVED` 상태인 주문에 대한 취소/수정 역시 없음** — `RESERVED` 주문에 대해 가능한
  동작은 승인과 거절뿐이다. 대기 중인 주문의 "수량/고객명 수정" 또는 "요청 철회" 플로우를
  만드는 것은 이번 이터레이션에서 명시적으로 범위 밖이다.
- 고객 대상 인터페이스 없음 — 고객은 주문 담당자가 입력하는 이름 문자열로만 표현되며, 별도의
  고객 요청 접수 채널을 만들지 않는다.
- 다중 생산 라인/병렬 생산 능력 없음 — 단일 FIFO 생산 큐만 범위에 포함되며, 동시 병렬 라인
  지원은 범위 밖이다. 따라서 어느 순간이든 "생산 중"인 큐 엔트리는 최대 하나이며, 나중에
  큐에 들어간 엔트리의 생산은 앞선 엔트리가 완료되기 전에는 시작될 수 없다.
- 세분화된 여유(sufficient)/부족(short) 재고 임계값 정책 없음 — 이 요구사항은 고갈
  (재고 == 0) 대 비고갈 구분만 확정한다; "N개 미만" 또는 "기준치의 X% 미만" 같은 문서화된
  충분성 임계값은 실제 사용 기준선이 마련된 이후의 향후 요구사항으로 미룬다. 지금 임의의
  임계값을 추측하여 구현하는 것은 명시적으로 범위 밖이다.
- `ORD-#####`(5자리) 롤오버 처리 없음 — 아래 주문 번호 관련 제약 참고; 이번 이터레이션에서
  10000번째 주문을 올바르게 처리할 필요는 없다.
- 더미 데이터 생성기나 데이터 모니터링 도구를 위한 별도 빌드 타겟/실행 파일 없음 — 이번
  이터레이션에서 둘 다 단일 `SampleOrderSystem` 콘솔 바이너리 안의 모드/메뉴로만 접근
  가능하다; 솔루션에 새 `.vcxproj` 타겟을 추가하는 것은 범위 밖이다.
- GUI 없음, 콘솔/터미널 인터페이스만 지원.

## 제약 사항 (Constraints)

- C++20, Visual Studio 프로젝트(`SampleOrderSystem.slnx`), 유니코드 문자셋, 콘솔 서브시스템,
  `msbuild SampleOrderSystem.slnx /p:Configuration=Debug /p:Platform=x64`로 빌드 가능해야
  한다(`CLAUDE.md` 기준).
- 모든 영속 상태(시료/재고, 주문, 생산 큐)는 애플리케이션이 직접 읽고 쓰는 JSON 파일이어야
  한다 — ORM 없음, 외부 DB 엔진 없음.
- JSON 파싱/파일 I/O는 `DataPersistence-dellacelina-1`의 방식을 따라야 한다; UI 구조는
  `ConsoleMVC-dellacelina-1`의 MVC 방식을 따라야 한다; 데이터 저장소 관리는
  `DataMonitor-dellacelina-1`을 따라야 한다; 더미 데이터 생성은
  `DummyDataGenerator-dellacelina-1`을 따라야 한다 — 이들은 그대로 베낄 코드가 아니라
  참고할 패턴이다.
- 더미 데이터 생성기와 데이터 모니터링 도구는 기존의 단일 `SampleOrderSystem` 콘솔
  실행 파일(`.vcxproj` 하나, 진입점 하나) 안의 추가 모드/메뉴 항목으로 구현한다. 이번
  이터레이션에서는 새로운 Visual Studio 프로젝트 파일이나 추가 `.exe` 빌드 타겟을 솔루션에
  추가하지 않는다.
- **평균 생산시간은 정수 분 단위로 표현하고 저장한다**(정수, 0보다 큼). 이 단위는 시료 스키마
  필드, 더미 데이터 생성기의 출력, 모든 완료 시각 계산에 적용된다.
- **시간 소스는 단일 주입 가능 시계(clock) 인터페이스로 추상화한다**(예: "현재 시각"을
  제공하는 `IClock`/`Clock` 추상화), 지연 정산이나 완료 시각 계산이 현재 시각을 필요로 하는
  모든 곳에서 사용한다. 프로덕션 코드는 실제 벽시계 구현을 사용하고, 자동화 테스트는 가짜/
  제어 가능한 구현을 사용하여, 실시간으로 sleep하지 않고도 시뮬레이션된 시계를 앞당겨 완료
  시각 정산을 테스트할 수 있어야 한다.
- 주문번호는 `ORD-####`(4자리 0-패딩 순번, 예: `ORD-0001`)이어야 하며, 전역적으로 증가하고
  유일해야 한다. **9999번째 주문을 넘는 롤오버 동작은 이번 이터레이션에서 명시적으로 범위
  밖이다**(비목표 참고): 시스템이 5자리 주문번호를 생성하거나 받아들일 필요는 없으며, 구체적인
  동작(필드 확장, 오류, 랩어라운드)도 여기서 규정하지 않는다. 주문량이 한계에 근접하면 향후
  요구사항에서 다시 다뤄야 한다.
- 주문 상태 전이는 다음으로 제한된다: `RESERVED` → `CONFIRMED`, `RESERVED` → `PRODUCING`,
  `RESERVED` → `REJECTED`, `PRODUCING` → `CONFIRMED`, `CONFIRMED` → `RELEASED`. 그 외 전이는
  유효하지 않다.
- 부족분/생산 계산: 부족분 = 주문 수량 − 승인 시점의 미선점 가용 재고(부족분 > 0일 때만);
  실제 생산 수량 = `ceil(부족분 / 수율)`; 큐 엔트리의 생산 소요 시간 = 평균 생산시간(분)
  × 실제 생산 수량.
- 생산 라인이 단일 순차 FIFO 자원이므로, 큐 엔트리의 예상 완료 시각은 자신의 등록 시각만이
  아니라 바로 앞선 엔트리를 기준으로 계산된다:
  `completion[i] = max(enqueue_time[i], completion[i-1]) + duration[i]`, 큐가 등록 당시
  비어 있었던 경우 `completion[0]`은 선행 항목이 없으므로
  `completion[0] = enqueue_time[0] + duration[0]`이 된다.
- 생산 완료를 위한 백그라운드 스레드/타이머는 허용되지 않는다; 정산은 주문 상태, 재고,
  생산 라인 상태를 다루는 모든 조회 안에서 동기적으로 인라인으로 이루어져야 한다.
- 현재 프로젝트에는 테스트 프레임워크가 연결되어 있지 않으므로 구현 과정에서 하나를
  도입해야 한다(`CLAUDE.md`의 Build 섹션 참고), `.slnx`에 추가되는 네이티브 Visual Studio
  테스트 프로젝트 형태로(예: MSTest Native, Google Test, vcpkg/NuGet을 통한 Catch2) — 이
  저장소는 어디에도 CMake가 관여하지 않으므로 CMake/CTest 기반 구성은 아니다.

## 수락 기준 (Acceptance Criteria)

- [ ] 시료 ID, 이름, 평균 생산시간(양의 정수 분), 수율(0 < 수율 ≤ 1)로 시료를 등록할 수 있다;
      중복된 시료 ID로 등록하면 오류로 거부되며 기존 시료는 변경되지 않는다.
- [ ] 시료 목록 화면은 등록된 모든 시료의 ID, 이름, 평균 생산시간(분), 수율, 현재 재고 수량을
      보여준다; 시료 ID로 검색하면 정확히 일치하는 시료만 반환한다(또는 없음); 이름으로
      검색하면 대소문자 구분 없이 해당 텍스트를 부분 문자열로 포함하는 시료만 정확히
      반환한다. 그 외 검색 필드(예: 수율, 평균 생산시간)는 지원할 필요가 없다.
- [ ] 등록된 시료 ID, 고객명, 양의 정수 수량으로 주문을 접수하면 고유한 `ORD-####` 주문번호
      (4자리, 순차 발급)를 가진 `RESERVED` 상태의 새 주문이 생성된다; 등록되지 않은 시료 ID나
      0 이하의 수량으로 주문을 접수하면 오류로 거부되고 어떤 주문도 생성되지 않는다.
- [ ] `RESERVED` 상태인 동안 주문을 수정하거나 취소하는 사용자 기능은 존재하지 않는다;
      `RESERVED` 주문에 노출되는 동작은 승인과 거절뿐이다.
- [ ] 승인 대기 화면은 현재 `RESERVED` 상태인 주문만 나열하며, 각 항목은 주문번호, 고객,
      시료, 수량을 보여준다.
- [ ] 미선점 재고(재고에서 해당 시료의 다른 `PRODUCING`/`CONFIRMED` 주문이 선점한 수량을
      뺀 값)가 전체 수량을 충당하는 `RESERVED` 주문을 승인하면, 생산 큐 엔트리를 생성하지
      않고 재고 변화도 없이 바로 `CONFIRMED`로 전환된다.
- [ ] 미선점 재고를 초과하는 `RESERVED` 주문을 승인하면, 부족분 = 수량 − 미선점 재고를
      계산하고, 그 부족분(전체 주문 수량이 아님)에 대해 정확히 하나의 생산 큐 엔트리를
      생성하며, 주문을 `PRODUCING`으로 전환하고, 예상 완료 시각과 함께 해당 엔트리를 생산
      큐 파일에 영속화한다.
- [ ] `RESERVED` 주문을 거절하면 `REJECTED`로 전환되며, 이후 어떤 모니터링, 승인, 생산 큐
      화면에도 해당 주문은 나타나지 않는다.
- [ ] 어떤 시료의 재고가 50이고 각각 100개씩인 두 건의 주문이 순차적으로(둘 다 승인 순서대로)
      승인될 때, 첫 번째 주문의 생산 큐 엔트리는 부족분 50을 요청하고 두 번째 주문의 엔트리는
      부족분 100을 요청한다(첫 번째 주문의 진행 중인 50을 이중으로 계산하지 않음) — 승인
      로직에 대한 스크립트/단위 테스트로 검증한다.
- [ ] 큐가 비어 있는 상태에서 고정된 가짜 시계 시각 `t0`에 엔트리 A가 등록되고 소요시간이
      `dA`(A의 평균 생산시간 × 실제 생산 수량)일 때, A의 예상 완료 시각은 정확히
      `t0 + dA`다. 이어서 다른 주문에 대한 엔트리 B가 A의 완료 이전인 가짜 시계 시각 `t1`에
      등록되고 자신의 소요시간이 `dB`일 때, B의 예상 완료 시각은 정확히
      `max(t1, t0 + dA) + dB`다 — 즉 `t1 <= A.completion`인 경우 반드시
      `A.completion + dB`이며 `t1 + dB`가 아니다 — 주입 가능한 가짜 시계를 사용한 스크립트/
      단위 테스트로 검증한다(실시간 sleep 없음).
- [ ] 생산 큐 엔트리에 저장된 완료 시각이(주입된 시계 기준으로) 지났을 때, {주문 상태, 재고
      수준, 생산 라인 상태} 중 어느 것을 다음에 조회하든 해당 엔트리를 큐 파일에서 제거하고,
      시료의 재고를 `ceil(부족분 / 수율)`(올림)만큼 증가시키며, 해당 주문을 `PRODUCING`에서
      `CONFIRMED`로 전환한다 — 폴링/백그라운드 스레드는 전혀 관여하지 않는다(타이머/스레드가
      시작되지 않고 정산이 오직 조회 처리 코드 경로 안에서만 일어남을 검증하며, 실시간
      sleep 대신 가짜 시계로 경과 시간을 결정적으로 시뮬레이션하여 확인한다).
- [ ] `CONFIRMED` 주문은 출고(release)될 수 있다: 주문이 `RELEASED`로 전환되고, 해당 시료의
      재고가 출고 수량만큼 감소한다; `CONFIRMED` 상태가 아닌 주문을 출고하려 하면 오류로
      거부되며 어떤 상태 변화도 일어나지 않는다.
- [ ] 모니터링의 주문 건수 화면은 정확히 `RESERVED`, `CONFIRMED`, `PRODUCING`, `RELEASED`에
      대한 건수를 보고하며, `REJECTED` 주문은 어떤 건수나 합계에도 절대 포함하지 않는다.
- [ ] 모니터링의 재고 화면은 시료별로 원본 재고 수량과, 재고가 정확히 0일 때는 "고갈", 그
      외에는 하나의 비고갈 라벨(예: "재고 있음")을 정확히 표시하는 상태 라벨을 보고한다;
      더 세분화된 "여유" 대 "부족" 구분은 이번 이터레이션에서 요구되지 않으며(비목표 참고),
      여기에 명시되지 않은 어떤 임계값으로도 채점되어서는 안 된다. 명시된 임계값이 없기
      때문이다.
- [ ] 생산 라인 화면은 FIFO 큐의 머리에 해당하며 "생산 중"으로 표시되는 엔트리 최대 하나
      (큐가 비어있지 않은 경우)와, 그 뒤에 등록 순서대로 대기 중인 나머지 엔트리들을, 각각
      시료, 부족 수량, 실제 생산 수량, 예상 완료 시각과 함께 나열한다; 이 화면은 동시에
      "생산 중"으로 표시되는 엔트리를 둘 이상 절대 보여주지 않는다.
- [ ] 모든 시료, 주문, 생산 큐 데이터는 디스크의 JSON 파일에 저장된다; 애플리케이션을
      중지했다가 다시 시작해도 모든 시료 정의, 주문 상태, 재고 수준, 미완료 생산 큐 엔트리가
      재시작 이전과 정확히 동일하게 유지된다(라운드트립 영속화 테스트가 통과한다).
- [ ] JSON으로 저장되는 각 데이터 타입(시료, 주문, 생산 큐 엔트리)마다 명시적이고 저장소에
      커밋된 스키마 정의(필드명, 타입, 필수/선택 여부, 제약)가 있으며, 영속화 계층은 로드 시
      이 스키마에 대해 검증한다; 잘못된 형식의 레코드는 다른 레코드를 조용히 손상시키는 대신
      오류를 보고하며 로드에 실패한다. 시료 스키마의 평균 생산시간 필드는 정수 분 단위로
      문서화된다.
- [ ] 더미 데이터 생성기는 동일한 `SampleOrderSystem` 실행 파일에서 접근 가능한 모드/메뉴로
      존재하며(별도 빌드 타겟 아님), 스키마에 부합하는 무작위 시료·주문 레코드를 생성하여
      JSON 저장소에 추가한다; 대화형 주문 접수/승인 메뉴 전체를 거치지 않고도 실행할 수
      있다.
- [ ] 데이터 모니터링 화면은 동일한 `SampleOrderSystem` 실행 파일에서 접근 가능한 모드/메뉴로
      존재하며(별도 빌드 타겟 아님), 호출된 시점의 시료/재고, 주문, 생산 큐 JSON 저장소의
      현재 디스크 상 내용을 보여준다(해당 호출 자체로 촉발된 지연 정산 결과를 반영한다).
- [ ] 코드베이스는 Model, View, Controller가 명확히 구분된 컴포넌트로 조직되며
      (`ConsoleMVC-dellacelina-1`의 구조를 참고), 각각 디렉터리/네임스페이스로 독립적으로
      식별 가능하고, View 코드는 직접 파일 I/O를 수행하지 않으며 Model 코드는 콘솔 I/O를
      수행하지 않는다.
- [ ] `SampleOrderSystem.slnx`에 연결된 네이티브 Visual Studio 테스트 프로젝트(예: MSTest
      Native, Google Test, Catch2 — CMake/CTest 구성이 아님)를 통해 실행 가능한 자동화
      테스트 스위트가 최소 하나 존재하며, 다음을 다룬다: 주문 승인 분기(재고 충분/부족),
      부족분/실제 생산 수량/생산 시간 계산, (위 FIFO 연쇄 규칙에 따른, 주입 가능 가짜 시계를
      사용한) 다중 엔트리 생산 큐 완료 시각 순서, 지연 큐 정산, JSON 영속화 라운드트립.
