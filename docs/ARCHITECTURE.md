# Architecture: S-Semi 반도체 시료 생산주문관리 시스템

Satisfies: [docs/REQUIREMENT.md](REQUIREMENT.md)

## Overview

The repository currently contains only an empty console skeleton (`SampleOrderSystem/SampleOrderSystem.vcxproj`, `SampleOrderSystem.slnx`) — no application source exists yet, so this is a from-scratch design, not a modification of existing modules. The system is a single-process C++20 console application. Business logic and presentation are separated by folder/namespace within the **existing** `SampleOrderSystem` project plus one new test project:

- **`SampleOrderSystem` (existing console project, unchanged as the sole `.vcxproj`/`.exe`)** — contains everything: JSON value/parser/writer, schema-validated JSON file persistence, domain models (Sample, Order, ProductionQueueEntry), repositories, services (order lifecycle, production queue/lazy settlement, monitoring, dummy data generation), the injectable `IClock` abstraction, the MVC presentation layer (Views/Controllers), and `main.cpp`. Source is organized into subfolders (`Core/`, `Json/`, `Persistence/`, `Models/`, `Repositories/`, `Services/`, `Views/`, `Controllers/`) purely for readability — see Key Design Decision #1 for why this is a single project rather than a `SampleOrderCore` static library, and the Open Question this raises.
- **`SampleOrderSystemTests`** (new test project — explicitly sanctioned by the requirement, unlike a second app/library project) — Catch2-based console test binary that references `SampleOrderSystem`'s compiled non-UI sources, covering approval branching (including that `Approve` settles before computing), shortfall/actual-quantity/duration math, FIFO completion-time chaining with a fake clock, lazy settlement, order-number recovery after restart, and JSON round-trip persistence.

All persistent state (samples/stock, orders, production queue) is stored as JSON files under a `data/` directory read/written exclusively through the persistence layer; each JSON "table" has a schema document committed under `schema/`. Every path that reads order status, stock, or production-line state — including `OrderService::Approve`, not only read-only "query" screens — funnels through a single `ProductionService::SettleDueEntries()` call before that data is used, so lazy reconciliation has exactly one implementation, driven by the injected clock rather than the wall clock directly.

## Components

### Core / Clock
- `SampleOrderSystem/Core/IClock.h` — abstract `IClock` with `Now() -> TimePoint`.
- `SampleOrderSystem/Core/SystemClock.h/.cpp` — production implementation wrapping `std::chrono::system_clock`.
- Tests use an in-memory `FakeClock` (in `SampleOrderSystemTests`) implementing `IClock`, whose time is advanced explicitly — no real sleeps, satisfying the acceptance criteria on deterministic FIFO/settlement tests.

### JSON layer (mirrors `DataPersistence-dellacelina-1`'s approach)
- `SampleOrderSystem/Json/JsonValue.h/.cpp` — a recursive variant type (`Null/Bool/Number/String/Array/Object`) with accessors and builders.
- `SampleOrderSystem/Json/JsonParser.h/.cpp` — text → `JsonValue`, throwing a `JsonParseException` on malformed input (never silently corrupting other records — required by the "잘못된 형식의 레코드는 ... 오류를 보고하며 로드에 실패한다" acceptance criterion).
- `SampleOrderSystem/Json/JsonWriter.h/.cpp` — `JsonValue` → pretty-printed text.

### Schema + persistence (mirrors `DataMonitor-dellacelina-1`'s "JSON as DB" approach)
- `schema/sample.schema.json`, `schema/order.schema.json`, `schema/production_queue.schema.json` — committed schema documents: field name, JSON type, required/optional, and constraints (e.g. `averageProductionTimeMinutes: integer, required, > 0`; `yield: number, required, 0 < yield <= 1`; `orderNumber: string, required, pattern ORD-####`).
- `SampleOrderSystem/Persistence/Schema.h` — in-memory schema descriptor structs parsed from the above files.
- `SampleOrderSystem/Persistence/SchemaValidator.h/.cpp` — validates a `JsonValue` record array against a `Schema` field-by-field; returns a typed error rather than throwing, so a repository can report "load failed" with a clear reason.
- `SampleOrderSystem/Persistence/JsonFileStore.h/.cpp` — generic "table" abstraction: load (parse + validate against schema), save (write to temp file + atomic rename), used by every repository below. This is the single place that touches the filesystem for domain data.

### Domain models
- `SampleOrderSystem/Models/Sample.h/.cpp` — `sampleId`, `name`, `averageProductionTimeMinutes` (int), `yield` (double, 0–1], `currentStock` (int); `ToJson`/`FromJson`.
- `SampleOrderSystem/Models/Order.h/.cpp` — `orderNumber` (`ORD-####`), `sampleId`, `customerName`, `quantity`, `status` (enum `Reserved/Confirmed/Producing/Released/Rejected`); `ToJson`/`FromJson`.
- `SampleOrderSystem/Models/ProductionQueueEntry.h/.cpp` — `orderNumber`, `sampleId`, `shortfallQuantity`, `actualProducedQuantity`, `enqueuedAt`, `expectedCompletionAt`; `ToJson`/`FromJson`.

### Repositories (wrap `JsonFileStore` for one table each)
- `SampleOrderSystem/Repositories/SampleRepository.h/.cpp` — `data/samples.json`; CRUD + `FindById` (exact) + `FindByNameSubstring` (case-insensitive) + stock mutation helpers.
- `SampleOrderSystem/Repositories/OrderRepository.h/.cpp` — `data/orders.json`; append-only creation, status transition updates, `NextOrderNumber()`.
  - **Restart-safe sequence derivation:** the requirement requires order numbers to be "전역적으로 증가하고 유일해야 한다" (globally increasing and unique), and the JSON round-trip acceptance criterion requires state to survive a restart exactly — so the sequence counter cannot live only in process memory. `OrderRepository`'s constructor/`Load()` scans every loaded `orderNumber` in `data/orders.json`, parses the numeric suffix, and initializes its in-memory next-sequence counter to `max(existing suffixes) + 1` (or `1` if the file is empty/absent). `NextOrderNumber()` then increments and formats that counter in memory for the remainder of the process's lifetime; the next restart re-derives it the same way. This makes `orders.json` itself the single source of truth for the sequence — no separate counter file is introduced, since the numbers are already persisted redundantly in every order record. Rollover past `ORD-9999` is explicitly out of scope per the requirement, so no wraparound/expansion logic is implemented.

### Services (business logic — no console or file I/O of their own; only call repositories)
- `SampleOrderSystem/Services/OrderService.h/.cpp` — `SubmitOrder`, `ListPendingApprovals`, `Approve`, `Reject`, `Release`.
  - `Approve(orderNumber, IClock& clock)` **first calls `ProductionService::SettleDueEntries(clock)`**, then computes **unclaimed stock** = `sample.currentStock − Σ(order.quantity for all other orders on this sample with status ∈ {Producing, Confirmed})` (see Key Design Decisions) against the now-current stock/order data, decides `Confirmed` vs `Producing`, and if `Producing`, delegates shortfall/duration math and enqueueing to `ProductionService` — reading the queue tail's `expectedCompletionAt` only after settlement has removed any already-due entries, so the FIFO chain calc for the new entry is never anchored on a stale (already-past) completion time. This settle-then-decide ordering is required because Approve reads exactly the state (order status, stock, production-line/queue state) that the requirement's lazy-settlement bullet says must trigger a sweep before use, on any of those three queries.
- `SampleOrderSystem/Services/ProductionService.h/.cpp` — `ComputeShortfall`, `ComputeActualQuantity` (`ceil(shortfall / yield)`), `ComputeCompletionTime` (FIFO chain rule against the current tail of the queue), `Enqueue`, and `SettleDueEntries(IClock&)` — the single lazy-reconciliation entry point: sweeps `ProductionQueueRepository`, removes entries whose `expectedCompletionAt <= clock.Now()`, increments the sample's stock by `actualProducedQuantity`, and flips the matching order `Producing → Confirmed`. `SettleDueEntries` is idempotent and cheap to call repeatedly (a no-op sweep if nothing is due), which is what lets `OrderService::Approve` call it unconditionally alongside the other three callers below without any coordination between them.
- `SampleOrderSystem/Services/MonitoringService.h/.cpp` — calls `SettleDueEntries` first, then returns status counts (`Reserved/Confirmed/Producing/Released`, `Rejected` always excluded) and per-sample stock + `Depleted`/`InStock` label (no finer threshold, per Non-goals).
- `SampleOrderSystem/Services/ProductionLineViewService.h/.cpp` — calls `SettleDueEntries` first, then returns "in production" (queue head, if any) plus the remaining FIFO tail, each with sample, shortfall, actual quantity, expected completion.
- `SampleOrderSystem/Services/DummyDataGenerator.h/.cpp` — schema-driven random `Sample`/`Order` generation (mirrors `DummyDataGenerator-dellacelina-1`), writing directly through the repositories; invocable without going through the interactive order menus.

### Console MVC layer (mirrors `ConsoleMVC-dellacelina-1`)
- `SampleOrderSystem/Views/*.h/.cpp` (`SampleView`, `OrderView`, `MonitoringView`, `ProductionLineView`, `DataMonitorView`) — pure console rendering of data already fetched by controllers; no file I/O.
- `SampleOrderSystem/Controllers/*.h/.cpp` (`MainMenuController`, `SampleController`, `OrderController`, `MonitoringController`, `DataMonitorController`, `DummyDataController`) — read console input, call services/repositories, hand results to Views; no direct JSON/file access.
- `SampleOrderSystem/main.cpp` — constructs a `SystemClock` and the repositories/services, then either (a) enters `MainMenuController`'s interactive loop, or (b) if invoked with a CLI flag (e.g. `--dummy-data`, `--data-monitor`), runs that mode directly and exits — satisfying "실행 가능... 대화형 메뉴 전체를 거치지 않고" for both the dummy generator and data monitor. Both modes are also reachable as ordinary menu items inside `MainMenuController` for users who prefer the interactive path.

### Build/test wiring
- **No new `.vcxproj`/build target is added for application code.** All of the above lives inside the existing `SampleOrderSystem.vcxproj`, organized by subfolder only. This is a deliberate reversal from an earlier draft of this design that proposed a `SampleOrderCore` static library project — see Key Design Decision #1 and the Open Question below for why that was rejected and what residual risk remains.
- `SampleOrderSystem.slnx` gains exactly one new project reference: `SampleOrderSystemTests\SampleOrderSystemTests.vcxproj` (Application-type, native VS test project) — this addition is explicitly sanctioned by the requirement's testing constraint ("네이티브 Visual Studio 테스트 프로젝트 형태로... `.slnx`에 추가되는"), independent of the "no new project files" sentence governing dummy-data/data-monitor, which applies only to those two features.
- `SampleOrderSystemTests` links against `SampleOrderSystem`'s compiled objects (or, if MSBuild project-to-project linking of an `Application`-type project's objects proves awkward in practice, compiles the same non-UI `.cpp` files a second time via a shared `.vcxitems`/property-sheet include list, or via `#include`d source in one test translation unit — the phase-plan stage should pick whichever avoids duplicate-symbol/link friction with a single-EXE constraint in force). It vendors Catch2 as **`catch_amalgamated.hpp` + `catch_amalgamated.cpp`** (the actively-maintained Catch2 v3 amalgamated distribution — not a single header; v2's single-header `catch.hpp` is the legacy, unmaintained line and should not be used) — no NuGet/vcpkg dependency required, keeping the existing MSBuild-only workflow intact.

## Data Flow / Interactions

**Sample registration/search:** `SampleController` reads input → `SampleRepository::Add` (rejects duplicate `sampleId` without mutating the existing record) → `SampleView` renders the list/search results via `FindById`/`FindByNameSubstring`.

**Order submission:** `OrderController` → `OrderService::SubmitOrder(sampleId, customerName, quantity)` → validates sample exists and `quantity > 0` (else rejects, no order created) → `OrderRepository::NextOrderNumber()` (derived at load time from the max existing `ORD-####` in `data/orders.json`, per Repositories above) + create with status `Reserved`.

**Approval (sufficient stock):** `OrderService::Approve(orderNumber)` → **`ProductionService::SettleDueEntries(clock)` runs first**, updating any orders/stock whose production had already completed → computes unclaimed stock for the order's sample using this now-current data → unclaimed ≥ quantity → order → `Confirmed`, no stock change, no queue entry.

**Approval (insufficient stock):** after the same settle-first step, unclaimed < quantity → `shortfall = quantity − unclaimed` → `ProductionService::ComputeActualQuantity(shortfall, sample.yield)` → `ComputeCompletionTime` using the current queue tail's `expectedCompletionAt` (guaranteed not-yet-expired because settlement already ran, or `Now()` if queue empty) → `Enqueue` persists the entry to `data/production_queue.json` → order → `Producing`. The two-orders-of-100-against-stock-50 acceptance scenario is handled entirely by the unclaimed-stock formula: after order 1 becomes `Producing` (claims 100), order 2's unclaimed stock is `50 − 100 → clamped to 0`, so its shortfall is the full 100. The settle-first step is what keeps this scenario correct if, between order 1's approval and order 2's approval, order 1's production had already finished — without settling, order 1 would still be double-counted as claiming stock even after its queue entry had matured into `Confirmed` stock, and separately the FIFO-chain scenario would risk anchoring order 2's completion-time calc on order 1's already-past `expectedCompletionAt`.

**Rejection:** `OrderService::Reject` → `Reserved → Rejected`; `Rejected` orders are simply excluded by every downstream query (`ListPendingApprovals`, `MonitoringService`, production-line view) via a status filter, not a separate table.

**Lazy settlement (any query, including Approve):** `MonitoringController`, `OrderController` (both the pending-list query *and* `Approve`), and `ProductionLineViewService` each begin their work by calling `ProductionService::SettleDueEntries(clock)`. That call reads `data/production_queue.json`, removes any entry with `expectedCompletionAt <= clock.Now()`, adds `actualProducedQuantity` to the sample's stock, flips the corresponding order `Producing → Confirmed`, persists both files, and only then does the caller's own read (or, for `Approve`, its unclaimed-stock computation and queue-tail read) proceed against the now-current data. This directly implements the requirement's "주문 상태, 생산 라인 상태, 재고 중 어느 것을 조회하든" (whichever of order status / production-line state / stock is queried next) trigger condition — `Approve` reads all three of those, so it is a caller like the others, not an exception to the rule.

**Release:** `OrderController` → `OrderService::Release(orderNumber)` → only valid if order is `Confirmed` (else rejected, no state change) → order → `Released`, sample stock decremented by the released quantity.

**Dummy data / Data monitor:** `main.cpp` CLI-flag branch or `MainMenuController` menu item → `DummyDataController`/`DataMonitorController` → `DummyDataGenerator` (writes via repositories) or direct repository reads rendered by `DataMonitorView` (which also triggers the same lazy settlement before display, so the monitor reflects current reality, per the acceptance criterion).

## Key Design Decisions

1. **Single `SampleOrderSystem` project (no new `SampleOrderCore` static library), plus one new `SampleOrderSystemTests` project.** An earlier draft of this design proposed splitting business logic into a new `SampleOrderCore` static library so `SampleOrderSystemTests` could link it without duplicating sources. That conflicts with the requirement's explicit constraint: "더미 데이터 생성기와 데이터 모니터링 도구는 기존의 단일 `SampleOrderSystem` 콘솔 실행 파일(`.vcxproj` 하나, 진입점 하나) 안의 추가 모드/메뉴 항목으로 구현한다. 이번 이터레이션에서는 새로운 Visual Studio 프로젝트 파일이나 추가 `.exe` 빌드 타겟을 솔루션에 추가하지 않는다." Read literally, "새로운 Visual Studio 프로젝트 파일... 추가하지 않는다" (no new VS project files added this iteration) is not scoped only to the dummy-data/data-monitor sentence that precedes it — it reads as a general iteration-wide constraint, with the test-project addition called out as a *separate, explicit* exception elsewhere in Constraints. A `SampleOrderCore.vcxproj` would be a second new project file with no such exception carved out for it. Rather than silently assume the constraint doesn't apply to a library (as opposed to an `.exe`) project, this design treats it as covered by the literal text and keeps all application code inside the existing single `.vcxproj`, organized into subfolders instead. **This is flagged as an Open Question below** because the constraint's intent is ambiguous (it may have meant only to ban a second *executable*, in which case a static library would be fine and would simplify test linking); implementation should get explicit sign-off before the phase-plan stage locks in the "compile shared sources into the test project" workaround described under Build/test wiring, which is more awkward than a library reference would be.

2. **Unclaimed stock = current stock − Σ(full order quantity) of that sample's `Producing`/`Confirmed` orders**, not `Σ(shortfall)`. This is what makes the acceptance-criteria's 50-then-100-then-100 example work: the first `Producing` order claims its entire requested quantity (100), not just the queued shortfall (50), because the eventual `Confirmed`→`Released` flow will need the full quantity. Alternatives considered: claiming only the shortfall (rejected — it under-reserves stock and would let order 2 wrongly see 50 unclaimed) or claiming full quantity only for `Confirmed` (rejected — contradicts the requirement text explicitly saying `Producing` also preempts stock).

3. **Single settlement entry point (`ProductionService::SettleDueEntries`)**, called at the top of every path that reads order status, stock, or production-line state — including `OrderService::Approve`, not just read-only monitoring/list queries — rather than duplicating the sweep-and-update logic in each caller. This directly satisfies "조회"(query) being interpreted uniformly across all three state kinds the requirement names, and makes the fake-clock-driven tests exercise one code path instead of four near-duplicates. Approve is included explicitly because it is, functionally, a query of exactly that state before it decides and mutates.

4. **Clock injected via constructor, not a singleton/static.** Every service that needs "now" (`ProductionService`, indirectly `MonitoringService`/`ProductionLineViewService`/`OrderService::Approve` through it) takes an `IClock&`. Production wiring in `main.cpp` passes a `SystemClock`; tests pass a `FakeClock` they control. This is what the constraint on deterministic time-travel testing requires — a singleton clock would make tests interfere with each other or require global mutable state.

5. **Custom `JsonValue`/`JsonParser`/`JsonWriter` instead of a third-party JSON library.** The constraint explicitly says to follow `DataPersistence-dellacelina-1`'s approach rather than "any JSON library," and the project currently has zero dependencies wired up (no vcpkg/NuGet config exists) — introducing one just for JSON would go against the "smallest design" principle and against the explicit pattern-following constraint.

6. **Production queue file's array order is the FIFO order.** No separate "position" or "sequence" field is stored; append-on-enqueue and in-order iteration on settlement/view are sufficient and match how `DataMonitor-dellacelina-1`-style JSON tables are typically kept simple.

7. **Catch2 vendored as the amalgamated `catch_amalgamated.hpp`/`.cpp` pair** inside a plain `Application`-type `.vcxproj`, rather than MSTest Native or a vcpkg-managed Google Test. Catch2 v3 (the maintained line) is not distributed as a true single header — only the legacy, unmaintained v2 line ships `catch.hpp` — so this design vendors the two-file v3 amalgamation rather than pinning to an outdated single-header release. This still avoids introducing a package manager to a repo that currently has none, while producing a native VS test project runnable via `msbuild`/VS Test Explorer, satisfying the constraint that rules out CMake/CTest.

8. **`OrderRepository::NextOrderNumber()` re-derives its counter from `data/orders.json` at load time (max existing suffix + 1), rather than persisting a separate counter file or keeping counter state only in memory.** Keeping it in-memory-only would reset to 1 on every restart and collide with existing orders, violating the "globally increasing and unique" constraint and the JSON round-trip acceptance criterion. A separate counter file was considered and rejected as an unnecessary second source of truth: `orders.json` already contains every issued number, so scanning it is sufficient and keeps persistence simple (Key Design Decision #5's spirit).

9. **CLI flags in `main.cpp` for dummy-data/data-monitor modes, in addition to menu entries.** The requirement only says these must be reachable "without going through the interactive order menus" — supporting both a flag and a menu item is the minimal way to satisfy that while keeping a single entry point/binary (no new `.vcxproj` target, per the constraint).

## Open Questions

- **Does the "no new Visual Studio project files this iteration" sentence in the requirement's Constraints ban a `SampleOrderCore` static-library project, or only a second `.exe`/build target?** This design currently assumes the broader (literal) reading and keeps all application code in the single existing `SampleOrderSystem.vcxproj`, accepting a more awkward test-linking approach as the cost (see Key Design Decision #1 and Build/test wiring). If the intent was narrower — banning only additional *executables*, not internal static libraries used solely for test linkage — a `SampleOrderCore` split would be simpler to implement and test. This should be confirmed before the phase-plan stage commits to either the shared-sources-compiled-twice approach or a library split.
- Whether the `data/` and `schema/` directories should be resolved relative to the executable's directory or the process's current working directory — matters for how `msbuild`-produced binaries under `x64/Debug/` locate their JSON files at runtime. Left for the implementation-plan stage to pin down (likely: resolve relative to the executable path so it's stable regardless of launch directory).
- Exact console UI copy/language (Korean vs English strings) is unspecified in the requirement; the design above is language-agnostic (Views own the text), so this can be decided per-phase without affecting Model/Controller contracts.
- Whether `SampleOrderSystemTests` should also assert on `SampleOrderSystem`'s Controllers/Views (UI-level tests) or stay scoped to non-UI services/repositories only — the requirement's acceptance criteria only name Core-level behaviors, so the phase plan can decide if UI gets any automated coverage beyond manual verification.
